# Расхождения реализации с базовым учебным заданием

Документ фиксирует решения, которые отличаются от текста индивидуальной работы. Все основные функции работы сохранены, расхождения сводятся к улучшениям архитектуры, расширению набора функций и косметике интерфейса.

## 1. MVVM вместо логики в MainWindow.xaml.cs

В работе описан вариант, где обработчики кнопок и методы загрузки/фильтрации находятся в `MainWindow.xaml.cs`.

В приложении используется лёгкий MVVM: состояние экрана и команды находятся в `MainViewModel`, а `MainWindow.xaml.cs` только создаёт зависимости и назначает `DataContext`.

Причина: такой подход лучше соответствует SOLID и CleanCode. Окно отвечает за отображение, ViewModel — за сценарии, сервисы — за внешние операции.

Функция проекта не меняется: пользователь по-прежнему видит список треков, фильтрует его, выбирает трек, запускает воспроизведение, сохраняет файл и видит историю.

## 2. Сервисы вместо методов окна

В работе файловые операции, загрузка треков и запуск воспроизведения описаны как методы главного окна.

В приложении эти операции вынесены в сервисы:

- `ITrackRepository` и `InMemoryTrackRepository` — загружают список треков;
- `IFileService` — проверяет существование файлов и копирует треки;
- `ISaveFileDialogService` — показывает диалог сохранения;
- `IOpenFileDialogService` — диалог открытия (для импорта локальных треков);
- `IConfirmationService` — диалог подтверждения (используется при удалении трека);
- `IAudioPlayerService` — управляет воспроизведением.

Причина: сервисы отделяют бизнес-логику от интерфейса и упрощают проверку кода юнит-тестами.

Функция проекта не меняется: алгоритмы фильтрации, проверки файлов, сохранения и истории остаются теми же по смыслу.

## 3. Встроенный MediaPlayer вместо внешнего проигрывателя

В работе прослушивание описано как запуск аудиофайла стандартным проигрывателем операционной системы через `Process.Start`.

В приложении используется встроенное воспроизведение через WPF `MediaPlayer`: пользователь может запустить трек, поставить его на паузу, остановить и видеть прогресс.

Причина: встроенный плеер лучше соответствует дизайну Figma, где центральная панель выглядит как полноценная панель управления выбранным треком.

Функция проекта сохраняется: выбранный аудиофайл проверяется на существование и воспроизводится. Отличается только способ воспроизведения.

## 4. Композитный репозиторий и постоянное хранилище пользовательских треков

В работе описан только захардкоженный список треков, формируемый при запуске.

В приложении захардкоженный список из 16 «эталонных» треков сохранён без изменений (`InMemoryTrackRepository`), но поверх него работает `IUserTrackStorage`. Реализация `JsonUserTrackStorage`:

- хранит пользовательские треки в JSON-файле `%LocalAppData%/MusicLibrary/userTracks.json`;
- автоматически создаёт каталоги `Music/` и `Covers/` рядом с JSON для аудиофайлов и обложек;
- умеет добавлять и удалять записи (а заодно физически чистит связанные файлы при удалении).

При запуске `CompositeTrackRepository` объединяет оба источника: сначала идут эталонные 16 треков, затем пользовательские. Если в JSON оказался конфликтующий Id — пользовательский трек получает новый, начиная с N+1.

Причина: такая схема позволяет пользователю расширять библиотеку без перекомпиляции и сохранять состояние между запусками.

Функция проекта не меняется по смыслу: список треков по-прежнему загружается один раз при запуске и используется для отображения и фильтрации.

## 5. Импорт треков (локально и по ссылке)

В работе функция «скачивания» сводится к копированию выбранного из библиотеки трека в выбранную пользователем папку через `SaveFileDialog`. Это поведение полностью сохранено в кнопке «Сохранить».

Дополнительно реализован отдельный импорт треков **в саму библиотеку** через кнопку «＋ Добавить трек» в шапке. Открывается модальное окно с двумя режимами:

- **«Из файла»** — выбор локального mp3/wav через `OpenFileDialog`, файл копируется в `%LocalAppData%/MusicLibrary/Music/` (`LocalFileImportRequest`).
- **«По ссылке»** — прямая ссылка на mp3/wav, скачивание через `HttpClient` с проверкой расширения/Content-Type и ограничением 50 МБ (`UrlImportRequest`).

Оба сценария обрабатываются единым сервисом `TrackImporter`. Все исключения внутри импортёра преобразуются в `ImportResult.Error`, чтобы окно никогда не зависало в состоянии «импортирую…».

Также реализована возможность удаления добавленных треков: справа в панели плеера у пользовательского трека появляется красная кнопка «Удалить трек». Перед удалением — диалог подтверждения. Захардкоженные 16 треков удалить нельзя — они отличаются по пути файла (лежат в папке приложения, а не в `%LocalAppData%`).

Причина: текст работы говорит «скачивание выбранной композиции», и в учебной интерпретации это копирование из библиотеки наружу — но без возможности добавить новый трек *в* библиотеку использование приложения ограничено только 16 захардкоженными композициями. Импорт делает приложение по-настоящему пригодным к использованию.

Функция проекта не меняется по смыслу: добавленные треки попадают в ту же `ObservableCollection<Track>` и используют те же алгоритмы фильтрации и проверки файлов, что и исходные.

## 6. Автоопределение метаданных и обложек

В работе предполагается, что для каждого трека в коде заранее заданы название, исполнитель и жанр.

Для импортируемых пользователем треков эти поля редко присутствуют корректно, поэтому добавлен каскад резолверов:

**Метаданные** — `DefaultMetadataResolver`:
1. Читает ID3 через TagLib# (`TagLibSharpTagReader`).
2. Очищает «грязные» суффиксы вида `Twenty One Pilots [drivemusic.me]` или `(zaycev.net)` через регулярку и чёрный список агрегаторов (`drivemusic`, `muzofond`, `mp3uk`, `zaycev` и др.).
3. Если поля пустые или артист был «грязным» — обращается к `MusicBrainzClient` (REST API MusicBrainz, обязательный User-Agent, rate-limit 1 запрос/сек).
4. Если жанр всё ещё пустой — берёт `primaryGenreName` из iTunes Search API.
5. Если ничего не помогло — пытается распарсить имя файла как `Artist - Title.mp3`.

**Жанр** — англоязычные значения сводятся к русским через `RussianGenreNormalizer`: словарь из 50+ пар (`rock` → «Рок», `electronic` → «Электроника», `metal` → «Метал», `post-punk` → «Постпанк», `phonk` → «Фонк» и т.д.). Незнакомые жанры сохраняются «как есть» с заглавной буквы. «Мусорные» жанры вида `drivemusic.me`, `muzofond` или `https://...` отбрасываются.

**Обложка** — `CompositeCoverResolver`:
1. APIC-кадр из ID3.
2. iTunes Search API (`ItunesCoverClient` берёт `artworkUrl100` и заменяет регуляркой на `600x600bb.jpg`).
3. `ProceduralCoverGenerator` — последний рубеж: рисует 512×512 PNG градиент с буквой исполнителя. Цвета и буква детерминированы по `artist+title`, чтобы у одного трека всегда была одна и та же заглушка.

Перед сохранением пользователь видит форму с найденными метаданными и обложкой и может их поправить вручную.

Все внешние HTTP-вызовы (MusicBrainz и iTunes) защищены отдельными per-request таймаутами по 10 секунд через `CancellationTokenSource`. Глобальный `HttpClient.Timeout` оставлен дефолтным, чтобы не мешать скачиванию большого аудиофайла по URL.

Причина: без автоопределения пользователю пришлось бы вводить все поля для каждого трека руками, что превратило бы приложение в утилиту учёта, а не музыкальный проигрыватель.

Функция проекта не меняется по смыслу: треки по-прежнему описываются классом `Track` со свойствами Title/Artist/Genre/Duration/FilePath, и используются по тем же алгоритмам.

## 7. Разделение «выделено» и «играет»

В первоначальной версии приложения сеттер `MainViewModel.SelectedTrack` безусловно вызывал `Stop()`, поэтому при смене жанра в фильтре или клике по другому треку воспроизведение тут же сбрасывалось.

В текущей версии состояние разделено на два:

- `SelectedTrack` — что показано на правой панели (обложка, информация, редактируемые поля).
- `PlayingTrack` — что сейчас играет в `IAudioPlayerService`.

`Stop` вызывается только когда пользователь нажимает «Стоп», стартует другой трек или текущий завершается/падает. Если выделен трек, который не играет — под обложкой появляется бейдж «▶ играет: …» с информацией о фактически играющем треке.

Причина: текст работы не описывает поведение конкретно (где пользователь выделил один трек, а играет другой), но в реальном использовании сброс воспроизведения при простой смене фильтра — раздражающий баг.

Функция проекта не меняется: воспроизведение, история, фильтрация, проверка файлов работают по тем же алгоритмам.

## 8. Реальные обложки для эталонной библиотеки

В первой версии в папке `MusicLibrary/Covers/` лежали процедурные PNG-заглушки 14–16 КБ каждая.

После того как был построен механизм автоматического получения обложек из iTunes (см. пункт 6), все 16 эталонных треков получили реальные обложки альбомов через тот же API:

- 13 точно совпавших (Benny Benassi, Dynoro, FIZICA, Кипелов, Molchat Doma, Mother Mother, Muse, MYTH & ROID, Onsa Media — Gods, Placebo, Powerwolf, Raon, The White Stripes);
- 3 заменены на обложки других треков того же артиста, потому что искомые в iTunes отсутствуют (KILLA! и KNIGHT от SHADXWBXRN, Soap Lagoon от Onsa Media). Стилистически у одного исполнителя обложки достаточно однородны, что лучше градиентной заглушки.

Расширение файлов обложек поменялось с `.png` на `.jpg`, в `InMemoryTrackRepository.cs` пути обновлены.

Функция проекта не меняется: обложки по-прежнему лежат локально в каталоге приложения и привязываются к треку через свойство `CoverPath`.

> **Примечание (релиз).** В релизной сборке набор «эталонных» треков сокращён до трёх для уменьшения размера установщика и из-за лицензионных ограничений: оставлены `Mother Mother — Hayloft II`, `MYTH & ROID — VORACITY` и `Кипелов — Я свободен`. Остальные 13 треков с обложками удалены из репозитория, но `InMemoryTrackRepository` и логика приложения с ними по-прежнему совместимы — пользователь может добавить треки самостоятельно через «＋ Добавить трек».

## 9. Косметические правки интерфейса

- Убран подзаголовок «Премиальная коллекция» в шапке — лишняя надпись.
- Добавлен `Padding` у центральной `ScrollViewer`, чтобы прогресс-бар плеера не упирался в полосу прокрутки в узком окне.
- ComboBox жанров получил полный `ControlTemplate` с тёмным popup, золотой обводкой, hover/selected состояниями. Раньше использовался дефолтный системный шаблон с белым popup.
- Текст выбранного жанра в закрытом ComboBox явно покрашен в светлый цвет через `TextElement.Foreground` (без этого ContentPresenter оборачивал строку в TextBlock с дефолтным чёрным цветом).
- В шапке справа появилась кнопка «＋ Добавить трек».
- В правой панели плеера появилась красная кнопка «Удалить трек», видимая только для пользовательских треков.

## 10. Первый пакет MusicBakh UX

В работе интерфейс описан как простая музыкальная библиотека с выбором трека, фильтром, воспроизведением, сохранением и историей.

В приложении первый UX-пакет оформляет этот же функционал как продукт MusicBakh:

- в шапке используется название MusicBakh и знак MB с аудиоволной;
- выбранный трек и фактически воспроизводимый трек визуально разделены;
- карточка текущего воспроизводимого трека подсвечивается в библиотеке независимо от выбранного фильтра;
- двойной клик по карточке трека запускает воспроизведение;
- двойной клик по элементу истории снова запускает этот трек;
- центральная панель плеера стала компактнее, чтобы основные элементы помещались без лишней прокрутки.

Причина: базовая работа описывает обязательные функции, но не фиксирует UX для ситуации, когда пользователь смотрит один трек, а играет другой. Разделение состояний делает приложение понятнее и готовит интерфейс к следующим пакетам: активному прогресс-бару, автопереходу к следующему треку и будущей NAudio-визуализации.

Функция проекта не меняется по смыслу: библиотека, фильтр, воспроизведение, сохранение и история остаются теми же учебными сценариями, но получают более ясное поведение и визуальную обратную связь.

## 11. Интерактивные элементы плеера

В работе описано базовое воспроизведение: выбрал трек — нажал кнопку — играет. Громкости, перемотки и режимов повтора в работе не предусмотрено.

Второй UX-пакет MusicBakh добавляет полноценную панель плеера:

- активный seek-слайдер с перетаскиванием и кликом по дорожке;
- кнопки перемотки на ±10 секунд;
- кнопки «предыдущий» и «следующий» — переход по видимому списку библиотеки;
- слайдер громкости (0–100%) и отдельная кнопка «без звука»;
- переключатель режима повтора: Off / Current (повтор трека) / Library (повтор всей видимой библиотеки);
- авто-переход к следующему треку после завершения текущего, с учётом режима повтора;
- горячие клавиши: Space — Play/Pause, ←/→ — перемотка ±10 секунд, Ctrl+←/→ — предыдущий/следующий трек, M — без звука, R — следующий режим повтора;
- сохранение громкости, состояния mute и режима повтора между запусками приложения (`%LocalAppData%\MusicBakh\player-settings.json`).

Известное ограничение: когда фокус находится на seek-слайдере, слайдере громкости или ListBox библиотеки/истории, клавиши ←/→ перехватываются этими контролами и до Window не доходят. Перенести их через `KeyDown + e.Handled` потребует code-behind — отнесено в следующий пакет.

Причина: базовая работа описывает «прослушать выбранный трек», но реальный плеер без перемотки и громкости неудобен. Управление воспроизведением — ожидаемый минимум для современного desktop-плеера, и без него UX из первого пакета (двойной клик, подсветка играющего трека) не доходит до конечного смысла.

Функция проекта не меняется по смыслу: библиотека, фильтр, воспроизведение, сохранение и история остаются теми же сценариями. Добавляются интерактивные элементы управления, которые относятся к UX, а не к новой функциональности.
